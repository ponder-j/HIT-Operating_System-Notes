# Chapter 6 中断和异常处理 - 读书笔记

## 3.1 中断和异常处理概述

### 什么是中断和异常？

**中断（Interrupts）**：
- 在程序执行期间随机发生，响应来自硬件的信号
- 系统硬件使用中断来处理处理器外部的事件，如外设服务请求
- 软件也可以通过执行 INT n 指令生成中断

**异常（Exceptions）**：
- 当处理器在执行指令时检测到错误条件时发生
- 包括保护违例、页错误、内部机器故障等
- 例如：除零错误、无效操作码等

### 处理器如何处理？

1. **中断/异常发生时**：
   - 暂停当前程序或任务的执行
   - 处理器执行中断或异常处理程序（interrupt/exception handler）
   - 使用向量号（vector number，0-255）作为 IDT 的索引

2. **处理完成后**：
   - 恢复被中断程序或任务的执行
   - 在没有程序连续性损失的情况下恢复（除非异常无法恢复或中断导致程序终止）

### 思考：实模式和保护模式下，中断向量表一样吗？

**不一样。**

- **实模式**：使用中断向量表（Interrupt Vector Table, IVT），位于内存地址 0x00000-0x003FF，每个表项4字节
- **保护模式**：使用中断描述符表（Interrupt Descriptor Table, IDT），每个表项8字节（64位模式为16字节），包含段选择子和偏移量，支持权限检查

## 3.2 有关中断和异常了解性的内容

### 中断和异常向量

**向量号范围**：0-255

**向量分配**：
- **0-31**：Intel 架构保留，用于架构定义的异常和中断
- **32-255**：用户定义的中断，通常分配给外部I/O设备

**常见异常向量**：
- 0: DE - 除法错误
- 1: DB - 调试异常
- 2: NMI - 不可屏蔽中断
- 3: BP - 断点
- 6: UD - 无效操作码
- 8: DF - 双重故障
- 13: GP - 通用保护故障
- 14: PF - 页故障

### 中断源和异常源

**中断源**：
1. **外部（硬件生成）中断**：
   - 通过处理器引脚或 local APIC
   - LINT[1:0] 引脚（可编程）
   - INTR 引脚（可屏蔽）
   - NMI 引脚（不可屏蔽）

2. **软件生成的中断**：
   - INT n 指令

**异常源**：
1. **处理器检测到的程序错误异常**
2. **软件生成的异常**：INTO, INT 3, BOUND 指令
3. **机器检查异常**：内部硬件错误和总线错误

### 异常的分类：故障、陷阱和中止

**故障（Faults）**：
- 通常可以被纠正，纠正后程序可以无损失地重新启动
- 返回地址指向**引起故障的指令**
- 例如：页故障 (#PF)、段不存在 (#NP)

**陷阱（Traps）**：
- 在执行陷阱指令后立即报告
- 返回地址指向**陷阱指令之后的指令**
- 允许程序继续执行而不损失连续性
- 例如：断点异常 (#BP)、溢出 (#OF)

**中止（Aborts）**：
- 不总是报告引起异常的指令的精确位置
- 不允许重新启动程序或任务
- 用于报告严重错误，如硬件错误
- 例如：双重故障 (#DF)、机器检查 (#MC)

### 程序或任务的重新执行

- **所有异常**（除中止外）保证在指令边界上报告
- **所有中断**保证在指令边界上接收

**故障类异常**：
- 返回指令指针指向故障指令
- 重新执行故障指令（常用于处理页故障）

**陷阱类异常**：
- 返回指令指针指向陷阱指令之后的指令
- 如果在传输执行的指令中检测到陷阱，返回指针反映传输

**中止类异常**：
- 不支持可靠的程序或任务重启
- 处理程序用于收集诊断信息并关闭系统

### 开启和禁止中断

**可屏蔽硬件中断的屏蔽**：
- **IF 标志**（EFLAGS 寄存器）：
  - IF = 0：禁止可屏蔽硬件中断
  - IF = 1：允许可屏蔽硬件中断
- **控制指令**：
  - STI：设置 IF 标志
  - CLI：清除 IF 标志
  - 仅在 CPL ≤ IOPL 时可执行

**IF 标志的影响**：
- 不影响 NMI（不可屏蔽中断）
- 不影响处理器生成的异常
- 通过中断门访问处理程序时自动清除 IF

**指令断点的屏蔽**：
- **RF 标志**：控制对指令断点条件的响应

**堆栈切换时的屏蔽**：
- MOV SS 或 POP SS 指令后，处理器禁止中断、调试异常和单步陷阱，直到下一条指令边界

### 异常和中断的优先级

当多个异常或中断在指令边界上挂起时，处理器按可预测的顺序处理它们：

| 优先级 | 描述 |
|--------|------|
| 1（最高）| 硬件复位和机器检查 |
| 2 | 任务切换上的陷阱 |
| 3 | 外部硬件干预（FLUSH, STOPCLK, SMI, INIT）|
| 4 | 前一条指令的陷阱（断点、调试陷阱）|
| 5 | 不可屏蔽中断（NMI）|
| 6 | 可屏蔽硬件中断 |
| 7 | 代码断点故障 |
| 8 | 取下一条指令的故障 |
| 9 | 解码下一条指令的故障 |
| 10（最低）| 执行指令的故障 |

## 3.3 中断描述符表（IDT）

### 如何构成？

**IDT 基本结构**：
- 8字节描述符数组（保护模式）
- 16字节描述符（64位模式）
- 最多256个描述符（向量0-255）
- 第一个条目可以包含描述符（与GDT不同）

**IDT 位置**：
- 可以位于线性地址空间的任何位置
- 基地址应在8字节边界上对齐以获得最佳性能
- 限制值 = 8N - 1（N 为描述符数量）

**IDTR 寄存器**：
- 48位寄存器
- 高32位：IDT 基地址
- 低16位：IDT 限制

### 如何获得中断处理程序的地址？

1. **向量号 × 8**（或 ×16 在64位模式）→ IDT 索引
2. **从 IDTR** 获取 IDT 基地址
3. **基地址 + 索引** → 门描述符地址
4. **从门描述符**读取：
   - 段选择子 → 指向 GDT/LDT 中的代码段描述符
   - 偏移量 → 处理程序在代码段中的偏移
5. **段描述符基地址 + 偏移量** → 处理程序的线性地址

### 如何设置中断描述符表寄存器？

**LIDT 指令**（加载 IDT 寄存器）：
- 格式：`LIDT m48`（或 m80 在64位模式）
- 从内存操作数加载基地址和限制到 IDTR
- **仅在 CPL = 0 时可执行**
- 通常由操作系统初始化代码使用

**SIDT 指令**（存储 IDT 寄存器）：
- 格式：`SIDT m48`
- 将 IDTR 的基地址和限制复制到内存
- 可在任何特权级执行

## 3.5 中断与异常处理

### 中断过程调用的流程

**完整流程**：

1. **接收中断/异常**：
   - 处理器检测到中断或异常条件
   - 确定向量号（0-255）

2. **访问 IDT**：
   - 向量号 × 8（或16）作为索引
   - 从 IDTR 获取 IDT 基地址
   - 读取门描述符

3. **权限检查**：
   - 检查 DPL（对于软件中断）
   - 检查段选择子的有效性
   - 检查目标代码段的 DPL

4. **判断是否需要堆栈切换**：
   - 比较目标代码段 DPL 与当前 CPL
   - 如果 DPL < CPL（更高特权级）→ 需要堆栈切换

5. **执行堆栈切换（如果需要）**：
   - 从当前 TSS 读取新的 SS 和 ESP
   - 在新堆栈上保存旧的 SS:ESP
   - 加载新的 SS:ESP

6. **保存处理器状态**：
   - 保存 EFLAGS
   - 保存 CS:EIP
   - 如果有错误代码，保存错误代码

7. **更新处理器状态**：
   - 如果是中断门，清除 IF 标志
   - 清除 TF、VM、RF、NT 标志

8. **转移控制**：
   - 加载新的 CS:EIP（从门描述符）
   - 开始执行中断/异常处理程序

9. **处理程序返回（IRET）**：
   - 恢复 CS:EIP
   - 恢复 EFLAGS
   - 如果有堆栈切换，恢复 SS:ESP

### 如何判断中断处理过程与被中断任务的优先级？

通过比较以下值：
- **当前特权级（CPL）**：由 CS 寄存器的低2位确定
- **目标代码段 DPL**：从门描述符指向的代码段描述符中获取

**判断规则**：
- 如果目标 DPL < CPL → 特权级提升（需要堆栈切换）
- 如果目标 DPL = CPL → 同特权级（不需要堆栈切换）
- 处理器不允许转移到特权级更低的代码段（DPL > CPL）

### 不同优先级上，处理方式一样吗？

**不一样。主要区别在于堆栈切换。**

**特权级改变（DPL < CPL）**：
- 需要堆栈切换
- 从 TSS 加载新的 SS:ESP
- 保存旧的 SS:ESP 到新堆栈
- 提供隔离的栈环境

**相同特权级（DPL = CPL）**：
- 不需要堆栈切换
- 使用当前堆栈
- 直接在当前堆栈上保存状态

### 如果发生堆栈切换，处理器会做哪些操作？

**详细步骤**：

1. **从 TSS 获取新堆栈指针**：
   - 根据目标 CPL，从 TSS 的 SS0/ESP0、SS1/ESP1 或 SS2/ESP2 读取
   - 临时保存旧的 SS:ESP

2. **加载新的 SS:ESP**：
   - 将新的段选择子加载到 SS
   - 将新的栈指针加载到 ESP

3. **在新堆栈上保存旧的栈指针**：
   ```
   PUSH SS    ; 旧的栈段
   PUSH ESP   ; 旧的栈指针
   ```

4. **在新堆栈上保存处理器状态**：
   ```
   PUSH EFLAGS
   PUSH CS
   PUSH EIP
   [PUSH Error Code]  ; 如果有错误代码
   ```

5. **堆栈布局**（从高地址到低地址）：
   ```
   +----------------+
   |      SS        |  ← 旧堆栈段
   +----------------+
   |      ESP       |  ← 旧堆栈指针
   +----------------+
   |    EFLAGS      |
   +----------------+
   |      CS        |
   +----------------+
   |      EIP       |
   +----------------+
   |  Error Code    |  ← ESP（如果有）
   +----------------+
   ```

### 如果没发生堆栈切换，处理器会做哪些操作？

**步骤**：

1. **使用当前堆栈**：
   - 不改变 SS:ESP

2. **在当前堆栈上保存处理器状态**：
   ```
   PUSH EFLAGS
   PUSH CS
   PUSH EIP
   [PUSH Error Code]  ; 如果有错误代码
   ```

3. **堆栈布局**（从高地址到低地址）：
   ```
   +----------------+
   |    EFLAGS      |
   +----------------+
   |      CS        |
   +----------------+
   |      EIP       |
   +----------------+
   |  Error Code    |  ← ESP（如果有）
   +----------------+
   ```

### 中断处理过程后，如何返回，处理器做了哪些操作？

**IRET 指令操作**：

1. **从堆栈弹出返回地址和标志**：
   ```
   POP EIP
   POP CS
   POP EFLAGS
   ```

2. **检查是否有特权级改变**：
   - 比较返回的 CS 的 CPL 与当前 CPL

3. **如果有特权级改变（返回到较低特权级）**：
   ```
   POP ESP    ; 恢复旧的堆栈指针
   POP SS     ; 恢复旧的堆栈段
   ```

4. **恢复 EFLAGS**：
   - **IOPL**：仅在 CPL = 0 时恢复
   - **IF**：仅在 CPL ≤ IOPL 时改变
   - 其他标志正常恢复

5. **继续执行被中断的程序**：
   - 从保存的 CS:EIP 处继续执行

### 异常和中断处理过程的保护

**保护机制**：

1. **特权级检查**：
   - CPL ≤ 门的 DPL（对于 INT n 指令）
   - 处理程序代码段 DPL ≤ CPL
   - 防止低特权级代码访问高特权级处理程序

2. **类型检查**：
   - 验证描述符类型正确（中断门、陷阱门或任务门）
   - 目标必须是代码段

3. **存在性检查**：
   - 检查 P 位（段存在位）
   - 不存在则产生 NP 异常

4. **段限制检查**：
   - 偏移量必须在段限制内
   - 超出则产生 GP 异常

5. **对齐检查**：
   - 堆栈必须正确对齐（64位模式中对齐到16字节）

### 异常和中断处理过程的标志使用方式

**EFLAGS 标志的处理**：

1. **TF（陷阱标志）**：
   - 在调用处理程序前自动清除
   - 防止指令跟踪影响中断响应
   - IRET 恢复

2. **IF（中断允许标志）**：
   - **中断门**：自动清除（禁止中断）
   - **陷阱门**：不影响
   - IRET 恢复（受 CPL 和 IOPL 限制）

3. **NT（嵌套任务标志）**：
   - 调用处理程序前清除
   - IRET 恢复

4. **VM（虚拟8086模式标志）**：
   - 调用处理程序前清除
   - IRET 恢复

5. **RF（恢复标志）**：
   - 调用处理程序前清除
   - IRET 恢复
   - 防止调试异常循环

6. **IOPL（I/O特权级）**：
   - 保持不变
   - 仅在 CPL = 0 时 IRET 可恢复

### 中断门与陷阱门的唯一区别是什么？

**唯一区别在于对 IF 标志的处理**：

- **中断门**：
  - 在进入处理程序时**自动清除 IF 标志**
  - 禁止其他可屏蔽中断
  - 防止中断嵌套
  - 适用于需要独占执行的中断处理程序

- **陷阱门**：
  - **不影响 IF 标志**
  - 允许其他中断发生
  - 适用于异常处理和调试

**其他方面完全相同**：
- 都保存 CS:EIP 和 EFLAGS
- 都进行特权级检查
- 都可以引起堆栈切换
- 都使用 IRET 返回
- 描述符格式相同（仅 Type 字段不同）

## 总结

### 关键概念

1. **中断和异常是处理器响应特殊条件的机制**
2. **使用向量号（0-255）索引 IDT 找到处理程序**
3. **异常分为故障、陷阱和中止三类**
4. **通过门描述符和特权级检查实现保护**
5. **堆栈切换机制隔离不同特权级的执行环境**

### 重要机制

- **IDT**：中断描述符表，操作系统的核心数据结构
- **门描述符**：中断门、陷阱门、任务门
- **堆栈切换**：特权级改变时的关键操作
- **标志管理**：IF、TF 等标志控制中断和调试行为

### 实际应用

- **页故障处理**：实现虚拟内存
- **系统调用**：用户程序请求操作系统服务
- **设备驱动**：响应硬件中断
- **异常处理**：错误恢复和调试
